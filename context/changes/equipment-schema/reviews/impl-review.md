### F1 — createClient returns null instead of throwing

- **Severity**: ⚠️ WARNING
- **Impact**: 🔎 MEDIUM — real tradeoff; pause to reason through it
- **Dimension**: Safety & Quality
- **Location**: src/lib/supabase.ts:7–9
- **Detail**: When SUPABASE_URL or SUPABASE_KEY are missing, createClient() returns null. Every call site must handle the nullable return (if (!supabase) return ...). A missing env var is a configuration error, not a runtime edge case — deferring failure to call sites means a misconfigured deployment fails silently until the specific code path is hit, rather than loudly at startup.
- **Fix**: Replace the null guard with an early throw: `if (!SUPABASE_URL || !SUPABASE_KEY) { throw new Error("SUPABASE_URL and SUPABASE_KEY must be set"); }` · Strength: Surfaces misconfiguration immediately with a readable message; removes null-check burden from every API route. · Tradeoff: If astro:env/server is declared with context: "server", Astro already enforces presence at build time — the throw is redundant but still safer. · Confidence: MEDIUM — depends on whether astro:env is already enforcing this at build time. · Blind spot: Haven't checked env.d.ts for astro:env declarations.
- **Decision**: FIXED — throw new Error(...) replaces null return; null guards removed from signup.ts, signin.ts, signout.ts, configs.ts, [id].ts (×2), dashboard.astro, middleware.ts

### F2 — handle_updated_at is a generic name in the public schema

- **Severity**: ⚠️ WARNING
- **Impact**: 🏃 LOW — quick decision; fix is obvious and narrowly scoped
- **Dimension**: Pattern Consistency
- **Location**: supabase/migrations/20260615000000_create_equipment_configs.sql:16
- **Detail**: CREATE OR REPLACE FUNCTION public.handle_updated_at() is declared inline in the equipment_configs migration with a generic name. If a future migration creates a second table and also calls CREATE OR REPLACE with any variation in the body, the two will silently collide — last write wins. Currently harmless but a maintenance trap.
- **Fix A ⭐ Recommended**: Add a SQL comment marking it as a shared utility: `-- Shared updated_at trigger function. Do not redefine inline in future migrations.` · Strength: Zero migration cost; immediately communicates intent to the next migration author. · Tradeoff: Relies on convention, not enforcement. · Confidence: HIGH — idiomatic Supabase approach. · Blind spot: None significant.
- **Fix B**: New migration that renames the function to equipment_configs_set_updated_at_fn. · Strength: Eliminates collision risk. · Tradeoff: Requires a second migration file; more churn for low-probability risk. · Confidence: MEDIUM.
- **Decision**: FIXED via Fix A — shared utility comment added above CREATE OR REPLACE FUNCTION

### F3 — equipment_type has no DB-level CHECK constraint

- **Severity**: OBSERVATION
- **Impact**: 🏃 LOW — quick decision; fix is obvious and narrowly scoped
- **Dimension**: Safety & Quality
- **Location**: supabase/migrations/20260615000000_create_equipment_configs.sql:6
- **Detail**: equipment_type TEXT NOT NULL DEFAULT 'dumbbell' accepts any string. S-03 added 'custom' as a 4th type — the absence of a CHECK constraint is what allowed that to work seamlessly. The app currently enforces the allowlist in both POST and PATCH API handlers, but there is no DB-level guarantee.
- **Fix**: Skip — app-layer validation (configs.ts and [id].ts both enforce the allowlist) is sufficient for a solo single-user app. Adding a CHECK now would need a migration update every time a type is added.
- **Decision**: FIXED — new migration 20260623000000_add_equipment_type_check.sql adds CHECK (equipment_type IN ('dumbbell', 'barbell', 'kettlebell', 'custom'))

### F5 — equipment_type typed as string, not a union

- **Severity**: OBSERVATION
- **Impact**: 🏃 LOW — quick decision; fix is obvious and narrowly scoped
- **Dimension**: Pattern Consistency
- **Location**: src/types/database.types.ts (Row/Insert/Update)
- **Detail**: equipment_type is `string` in the generated types. S-03 defined TypeMode = "dumbbell" | "barbell" | "kettlebell" | "custom" locally in components but doesn't flow from the DB type. The TypeMode cast in ConfigCard.tsx (config.equipment_type as TypeMode) is the correct workaround for a generated file.
- **Fix**: Skip — regenerating the types file would overwrite any manual union annotation. The TypeMode cast pattern is appropriate here.
- **Decision**: FIXED — exported EquipmentType union added to database.types.ts; equipment_type fields in Row/Insert/Update now use it; redundant `as TypeMode` cast removed from ConfigCard.tsx

### F6 — seed.sql uncommitted-state risk

- **Severity**: OBSERVATION
- **Impact**: 🏃 LOW — quick decision; fix is obvious and narrowly scoped
- **Dimension**: Safety & Quality
- **Location**: supabase/seed.sql
